# kol_lens 相对最终产品设计的待完善功能清单

## 结论摘要

基于当前仓库中已经落地的能力，以及路线图、后端逻辑摘要和本次新增后端服务的实现范围来看，项目已经具备了**可运行的第一版后端主链路**：自然语言需求理解、库内检索、结果不足时的外部扩库、外部达人入库、冷启动向量化，以及初版贪心降级都已经具备雏形。

但如果以“最终产品设计”的标准衡量，当前状态仍然更接近**后端 MVP + 产品主链路打通版**，距离真正完整的 Σ.Match 产品还有一段明显差距。核心缺口不在“有没有接口”，而在于**多模态生产能力、异步实时体验、反馈进化闭环、资产管理闭环，以及上线级工程能力**还没有全部补齐。

## 当前已完成与仍缺失的边界

| 模块 | 当前状态 | 结论 |
|---|---|---|
| 自然语言解析 | 已支持数据要求、风格要求、query plan、elastic weights | **已具备可用基础版** |
| 库内匹配 | 已支持 Milvus 检索与基础结果组织 | **已具备可用基础版** |
| 结果不足外部扩库 | 已支持 payload 生成、外部请求、入库、冷启动回查 | **已具备可用基础版** |
| 贪心降级 | 已有初版同步降级逻辑与日志 | **已具备 MVP 版** |
| 多模态特征流水线 | 仍以文本冷启动向量替代真实视觉/文本多模态链路 | **关键缺口** |
| 异步任务与实时推送 | 任务状态存储已存在，但仍是同步执行，缺 Celery / WebSocket 真正联动 | **关键缺口** |
| 反馈进化 | 尚未形成 Rocchio 或多轮选中/淘汰后的新一轮进化检索闭环 | **关键缺口** |
| 资产入库与历史管理 | 数据层有基础，但产品级 API 闭环仍未补齐 | **关键缺口** |
| 智能导出 | 数据表和字典能力存在，但业务接口未完成 | **明显缺口** |
| 前后端联调与上线能力 | 尚未完成 E2E、部署编排、监控告警、性能调优 | **上线前缺口** |

## 按最终产品设计仍缺少的重点功能

### 一、真实多模态特征提纯流水线仍未完成

这部分是我认为**当前最大缺口**。现在外部扩库后的达人虽然可以快速入库并参与召回，但使用的是文本冷启动向量，更多是为了先跑通产品链路，而不是最终设计中的多模态生产方案。按照路线图，最终版本需要图片下载、预处理、YOLO 人物/背景分割、PaddleOCR 文本提取、InsightFace 人脸向量、CLIP 场景向量、CLIP 文本向量，以及近 30 天笔记的时序融合，最后生成正式的 `v_face`、`v_scene`、`v_overall_style`。这一整段目前并没有真正落地。

这意味着现在的召回质量更多依赖文本近似和规则拼装，适合验证业务逻辑，但还不能代表最终产品的视觉风格匹配能力。若你要达到“按图片风格、人物气质、场景内容稳定寻人”的最终效果，这条链路必须尽快补齐。

### 二、排序与推荐引擎仍停留在基础检索阶段

当前版本已经能完成召回，但“最终产品设计”里要求的不只是召回，而是**召回之后的精排**。路线图里明确写了混合排序公式，需要结合余弦相似度、数据归一化、商业性价比和业务超参数做排序。目前仓库内主服务更偏“检索编排器”，还不是完整的“推荐引擎”。

更进一步，最终设计中的**Rocchio 向量平移**也还没有形成产品级闭环。也就是说，用户在评审时对达人做出“选中 / 待定 / 淘汰”之后，系统还不能基于这些反馈自动生成下一轮更精准的检索向量并真正发起“换一批”进化检索。这是 Σ.Match 差异化能力里的核心部分，目前仍属于待实现状态。

### 三、异步任务队列与实时推送还没有真正落成

目前的检索服务虽然已经有 `task_id`、任务状态存储和日志结构，但从实现形态上看，`submit_retrieve_task` 仍然是**同步执行后再写状态**，并不是路线图定义中的真正异步任务编排。最终设计要求 Celery Worker、Redis Broker、耗时任务后台执行、前台轮询或 WebSocket 实时推送。

这部分缺失会直接影响产品体验。尤其是当后面接入真实外部扩库、GPU 特征提取、多轮降级后，同步接口很容易超时，也无法把“降级过程”“扩库进度”“特征提取进度”实时展示给前端的终端组件。因此，**调度层从伪异步升级为真异步**，是进入产品可发布阶段前必须补的一环。

### 四、弹性降级还缺少产品级可观测性与完整策略树

当前已经有初版贪心降级，但距离最终设计仍有三点差距。第一，降级过程目前主要是同步日志回传，还没有真正接入实时终端推送。第二，降级逻辑虽然具备“放宽条件”的核心思想，但还没有完全产品化为明确的约束松弛树和优先级管理机制。第三，降级后的每一步影响范围、最终命中率、回退原因等可观测指标还没有标准化。

换句话说，现在的降级更像“算法层能工作”，但还不是“产品层可解释、可回放、可监控”的版本。最终产品如果想让策略、运营和销售团队都敢用，这一层要进一步结构化。

### 五、资产入库闭环仍未完成为正式产品接口

最终设计中，检索只是前半段，后半段还有**确认入库、品牌偏好沉淀、履约记录、资产库检索、达人历史查询**这些闭环动作。当前数据库层已经有相应基础，例如 `campaign_history`、`fulfillment_records`、达人与笔记数据表都存在，但 API 层并没有完整暴露成最终产品设计中的那组接口。

目前仍明显缺少的产品级接口包括以下几类：

| 缺失接口/能力 | 当前情况 | 影响 |
|---|---|---|
| `POST /api/v1/assets/commit` | 数据层可支撑，API 层未完成 | 无法完成“评审后确认入库”的正式闭环 |
| 选中/淘汰/待定反馈沉淀 | 还未形成完整产品接口与角色权重策略 | 无法支撑品牌偏好长期学习 |
| `GET /api/v1/library/list` | 数据层有查询基础，产品接口未完成 | 前端资产库无法稳定接真实库 |
| `GET /api/v1/library/history` | 历史查询数据层已具备，接口层未完成 | 无法展示达人履约/合作历史 |
| 履约时间轴 | 表结构已存在，但产品能力未接通 | 资产库价值没有真正体现出来 |

这部分如果不补齐，当前产品仍然是“智能找人系统”，而不是“寻星到资产沉淀的一体化工作站”。

### 六、智能导出能力尚未完成产品化

从数据库封装看，`export_dictionary` 相关的数据结构和映射沉淀能力已经有基础，但最终设计要求的是完整的**智能表头匹配与字典沉淀接口**，并服务前端的导出四步流程。当前这部分仍未对外形成正式接口，也没有与前端导出组件打通。

如果你的最终产品包含“客户导出、字段标准化、历史字典记忆”这一块，那么它仍属于明显待开发项，而不是简单补点样式或参数就能完成的功能。

### 七、前端联调仍未完成最终产品闭环

从最终设计看，前端不是只接一个检索接口，而是要串起以下链路：对话输入、意图确认、实时降级终端、数据矩阵浏览、沉浸式评审、换一批、确认入库、资产库、历史抽屉、智能导出。当前后端虽然已经补出了一部分关键接口，但距离前端全部替换 Mock 数据还有差距。

这意味着接下来真正的工作不会只是继续写几个后端模块，而是要按页面逐项联调，确保状态机、轮询或推送机制、结果结构、日志展示、异常处理都和前端组件契合。

### 八、基础设施与上线能力还不够生产级

当前测试结果已经表明，与本次新增链路直接相关的单测是可通过的，但基础设施集成测试仍依赖真实 PostgreSQL、Milvus、Redis 环境。这本身不是错误，但说明系统距离生产可用还差一层“环境级集成”。

按照最终设计，后续还需要补齐 Docker Compose 一键部署、环境变量与密钥治理、日志采集、监控告警、数据备份、Milvus 索引优化、Redis 缓存命中率优化、E2E 测试和性能压测。这些内容不补齐，系统可以开发演示，但还不适合稳定上线。

## 我建议的优先级顺序

如果目标是尽快把产品推到“真正能试用”的阶段，我建议优先级按下面顺序推进。

| 优先级 | 建议先做什么 | 原因 |
|---|---|---|
| P0 | 完成 `assets/commit`、`library/list`、`library/history`、`export/confirm-mapping` | 先把产品 API 闭环补齐，前端才能完整联调 |
| P0 | 把同步任务改成 Celery + Redis 真异步，并补 WebSocket 推送 | 这是产品体验从“能跑”到“能用”的关键一步 |
| P1 | 落地 Rocchio 反馈进化与多轮“换一批” | 这是产品核心差异化能力 |
| P1 | 完成正式多模态向量流水线，替换文本冷启动向量 | 决定匹配质量天花板 |
| P1 | 完成精排与业务评分模型 | 决定结果是否真正“好用” |
| P2 | 打通资产库前端、履约历史、智能导出全链路 | 决定资产沉淀与复用价值 |
| P2 | 做 E2E、部署、监控、性能优化 | 决定是否具备上线条件 |

## 一句话判断

如果你问我“现在离最终产品设计还差什么”，我的判断是：

> **现在已经把“找人主链路”做到了可用 Demo 级，但距离“可持续运营的完整产品”还缺四大块：真实多模态、真异步实时调度、反馈进化闭环、资产管理闭环。**

如果你愿意，我下一步可以继续直接帮你把这份缺口清单进一步拆成一个**按周排期的开发计划**，或者直接从优先级最高的 `assets/commit + library/list + library/history` 这一组接口开始继续落代码。
