# Σ.Match 智能寻星工作站 — 全栈开发路线图

> 本文档基于《创意逻辑/用户洞察/落地建议（极深全图景拆解）》中的五层架构设计与《前台产品需求与交互规范 (PRD)》中的页面交互定义，将整个系统拆解为 **6 个开发阶段（Sprint）**，每个阶段内部按**自底向上**的依赖关系排列，确保每一步都有可验证的交付物。

---

## 全局架构总览

Σ.Match 的技术架构可以抽象为五个垂直层次，从底层基建到顶层触点依次为：基建层、解剖层、大脑层、调度层、触点层。开发顺序遵循**"先铺管道，再装引擎，最后接仪表盘"**的原则——先确保数据能存、能取、能算，再让前端能看、能用、能反馈。

| 层次 | 定位 | 核心技术栈 | 对应 Sprint |
|------|------|-----------|------------|
| 基建层 | 数据存储与 Schema | PostgreSQL + Milvus + Redis | Sprint 1 |
| 解剖层 | 多模态特征提纯 | YOLO + PaddleOCR + InsightFace + CLIP | Sprint 2 |
| 大脑层 | 向量运算与推荐 | Milvus Hybrid Search + Rocchio 算法 | Sprint 3 |
| 调度层 | 贪心降级与并发 | Celery + Redis Queue + WebSocket | Sprint 4 |
| 触点层 | 前端交互与意图转译 | React + LLM Intent Parser | Sprint 5 |
| 全链路 | 联调、优化与上线 | E2E Testing + 性能调优 | Sprint 6 |

## Sprint 1：基建层 — 数据地基与存储架构（预计 2 周）

本阶段的目标是搭建整个系统的数据骨架。没有稳固的存储层，上层的向量运算和异步调度都是空中楼阁。

### 1.1 PostgreSQL 关系型数据库

首先建立核心业务表。`influencer_basics` 表存储达人的基础结构化信息（小红书号、昵称、头像、粉丝数、点赞收藏数、最新笔记时间戳等），其中 `red_id` 建立唯一索引，`latest_note_time` 用于后续触发特征更新机制。`campaign_history` 表存储品牌偏好沉淀数据，包含品牌名、SPU 名、已选中/已淘汰达人 ID 列表（JSONB 格式），这张表是 Rocchio 向量平移算法的历史数据来源。

| 任务编号 | 任务描述 | 验收标准 |
|---------|---------|---------|
| 1.1.1 | 设计并创建 `influencer_basics` 表 | 可插入、查询达人基础数据，唯一索引生效 |
| 1.1.2 | 设计并创建 `campaign_history` 表 | JSONB 字段可正确存取达人 ID 数组 |
| 1.1.3 | 设计并创建 `export_dictionary` 表（智能导出字典） | 支持非标准表头到标准字段的映射存取 |
| 1.1.4 | 编写 Migration 脚本与 Seed 数据 | 一键初始化开发环境 |

### 1.2 Milvus 向量数据库

创建 `influencer_multimodal_vectors` Collection，这是混合检索的核心。Schema 需要包含：`id`（INT64 主键，映射 PostgreSQL 的 `internal_id`）、`followers` 和 `region`（标量索引字段，用于硬筛选过滤）、`v_face`（512 维 FLOAT_VECTOR，存储 InsightFace 人脸特征）、`v_scene`（768 维 FLOAT_VECTOR，存储 CLIP 场景特征）、`v_overall_style`（768 维 FLOAT_VECTOR，时序融合后的最终风格向量）。标量字段的冗余存储是为了实现标量过滤与向量检索在数据库底层的闭环（Hybrid Search），避免跨库 JOIN 带来的延迟。

| 任务编号 | 任务描述 | 验收标准 |
|---------|---------|---------|
| 1.2.1 | 部署 Milvus 实例（Docker Compose） | 服务启动，可通过 pymilvus 连接 |
| 1.2.2 | 创建 Collection 并定义 Schema | 标量索引 + 向量索引均可查询 |
| 1.2.3 | 编写 Hybrid Search 基础查询封装 | 给定标量条件 + 向量，返回 Top-K 结果 |

### 1.3 Redis 缓存层

部署 Redis 实例，定义缓存键命名规范。主要用途包括：异步任务状态缓存（Task_ID → 进度/结果）、达人数据热缓存（减少 PostgreSQL 查询压力）、以及前端轮询的中间结果暂存。

| 任务编号 | 任务描述 | 验收标准 |
|---------|---------|---------|
| 1.3.1 | 部署 Redis 并定义键命名规范 | 连接正常，TTL 策略生效 |
| 1.3.2 | 封装任务状态读写工具函数 | set_task_status / get_task_status 可用 |

### Sprint 1 交付物

完成后应具备：三套存储服务全部就绪，可通过脚本插入测试数据并执行基础的关系查询、向量检索和缓存读写。

## Sprint 2：解剖层 — 多模态特征提纯流水线（预计 3 周）

本阶段构建 DAG（有向无环图）流水线，将达人的图片和文本内容转化为可计算的向量。这是整个推荐系统的"感官器官"。

### 2.1 数据采集节点（Node 1: Data Ingest）

编写图片下载与预处理模块。从小红书图床下载达人笔记图片，统一 Resize 到 224×224（CLIP 标准输入尺寸），存入内部 OSS（对象存储）。需要处理的边界情况包括：图片 404、下载超时、格式异常等，均需记录日志并跳过，不阻塞流水线。

| 任务编号 | 任务描述 | 验收标准 |
|---------|---------|---------|
| 2.1.1 | 图片下载器（支持并发、重试、超时） | 100 张图片下载成功率 > 95% |
| 2.1.2 | 图片预处理（Resize + 格式标准化） | 输出统一 224×224 JPEG |
| 2.1.3 | OSS 存储封装（上传/读取/删除） | 图片可通过 URL 访问 |

### 2.2 语义分割节点（Node 2: Semantic Split）

部署 YOLO 模型进行图像分割，输出两张掩码图：`Mask_Person`（人物区域）和 `Mask_Background`（背景/场景区域）。同时部署 PaddleOCR 提取图片中的文字内容 `Text_Content`，用于后续的文本向量化。

| 任务编号 | 任务描述 | 验收标准 |
|---------|---------|---------|
| 2.2.1 | 部署 YOLO 分割模型（GPU 推理） | 输入图片，输出 Person/Background 掩码 |
| 2.2.2 | 部署 PaddleOCR 文字提取 | 输入图片，输出识别文本 |
| 2.2.3 | 编写 Node 2 编排逻辑 | 一张图片同时产出掩码 + 文本 |

### 2.3 并行嵌入节点（Node 3: Parallel Embedding）

这是计算密集型的核心节点，需要 GPU 并行处理三条线程。Thread A 将 `Mask_Person` 覆盖的原图送入 InsightFace，输出 512 维人脸特征向量 `V_face`。Thread B 将 `Mask_Background` 覆盖的原图送入 CLIP Vision Model，输出 768 维场景特征向量 `V_scene`。Thread C 将 `Text_Content` 送入 CLIP Text Model，输出 768 维文本特征向量 `V_text`。三条线程的结果汇合后写入 Milvus。

| 任务编号 | 任务描述 | 验收标准 |
|---------|---------|---------|
| 2.3.1 | InsightFace 模型部署与推理封装 | 输入人脸图，输出 512 维向量 |
| 2.3.2 | CLIP Vision 模型部署与推理封装 | 输入场景图，输出 768 维向量 |
| 2.3.3 | CLIP Text 模型部署与推理封装 | 输入文本，输出 768 维向量 |
| 2.3.4 | GPU 并行编排（ThreadPoolExecutor） | 三线程并行，单图处理 < 2s |

### 2.4 时序融合节点（Node 4: Temporal Fusion）

实现时序衰减注意力机制。维护每个达人近 30 天笔记的滑动窗口（Sliding Window），新笔记进入时旧笔记权重呈指数级衰减。融合公式将 `V_face`、`V_scene`、`V_text` 按时间加权合成为最终的 `V_overall_style` 向量，写入 Milvus 的对应字段。

| 任务编号 | 任务描述 | 验收标准 |
|---------|---------|---------|
| 2.4.1 | 实现指数衰减权重函数 | 30 天前的笔记权重趋近于 0 |
| 2.4.2 | 实现滑动窗口融合逻辑 | 新笔记触发向量更新，旧笔记自动降权 |
| 2.4.3 | 将融合向量写回 Milvus | `v_overall_style` 字段更新成功 |

### Sprint 2 交付物

完成后应具备：给定一个达人的小红书主页链接，流水线可自动下载图片、分割、提取特征、融合并存入向量库。可通过脚本批量处理 100 个达人验证端到端流程。

## Sprint 3：大脑层 — 向量运算与推荐排序（预计 2 周）

本阶段实现推荐系统的核心数学逻辑，让系统具备"理解"和"进化"的能力。

### 3.1 混合排序计分函数（Hybrid Ranking）

当 Milvus 召回 Top-100 候选达人后，在内存中执行精排。计分公式融合四个维度：余弦相似度（衡量视觉和标签匹配度）、数据归一化得分（对赞藏评数据进行 Min-Max 归一化映射到 [0,1]）、商业性价比函数（基于历史商单报价计算 ROI）、以及业务超参数（由前端传入，决定本次搜索侧重"颜值"、"爆文率"还是"性价比"）。

| 任务编号 | 任务描述 | 验收标准 |
|---------|---------|---------|
| 3.1.1 | 实现余弦相似度计算模块 | 输入两个向量，输出 [-1, 1] 得分 |
| 3.1.2 | 实现 Min-Max 归一化模块 | 批量数据归一化到 [0, 1] |
| 3.1.3 | 实现性价比计算函数 | 基于报价和互动数据输出 ROI 指标 |
| 3.1.4 | 组装 Hybrid Ranking 主函数 | 输入 100 候选人，输出按分数排序的 Top-K |

### 3.2 Rocchio 向量平移（意图进化核心）

这是系统最关键的"黑科技"。当用户在前端点击"选中"和"淘汰"后，系统需要自动生成更精准的下一批推荐。具体操作为：从 Milvus 中取出被选中博主组（S）的平均特征向量，乘以正向权重 β（默认 0.75）；取出被淘汰博主组（R）的平均向量，乘以负向权重 γ（默认 0.25）。将原始查询向量 `V_original` 朝着 S 的方向拉近，同时从 R 的方向推远，生成新的查询向量 `V_next_query`，再次送入 Milvus 进行检索。

| 任务编号 | 任务描述 | 验收标准 |
|---------|---------|---------|
| 3.2.1 | 实现向量平均计算（选中组/淘汰组） | 正确计算多个向量的均值 |
| 3.2.2 | 实现 Rocchio 公式核心逻辑 | V_next 方向正确偏移 |
| 3.2.3 | 集成到检索流程（retrieve → rank → evolve） | "换一批"结果与前一批有明显差异化 |

### 3.3 反向打标与权重沉淀

当用户确认入库时，系统需要将选中达人的特征反向沉淀为品牌偏好。不同角色的选择权重不同：客户选择权重最高，策划/前端中等，采购基础。将加权后的特征向量更新到 `campaign_history` 表中，作为该 SPU 未来检索的基准向量。

| 任务编号 | 任务描述 | 验收标准 |
|---------|---------|---------|
| 3.3.1 | 实现角色权重映射逻辑 | 客户 > 策划 > 采购 权重正确 |
| 3.3.2 | 实现品牌偏好向量沉淀 | 入库后 campaign_history 正确更新 |

### Sprint 3 交付物

完成后应具备：给定一组筛选条件和用户反馈（选中/淘汰列表），系统可返回排序后的推荐结果，并在"换一批"时产生明显进化的新结果。

## Sprint 4：调度层 — 异步任务与弹性降级（预计 2 周）

本阶段是系统的"心脏起搏器"，确保前端体验流畅、后端不阻塞。

### 4.1 Celery 异步任务队列

搭建 Celery Worker 集群，将耗时操作异步化。Worker 1 负责 Milvus 向量库混合检索（预计耗时 0.5s），Worker 2 负责组装 Payload 访问小红书接口（若需实时抓取，耗时 2-5s）。前台请求到达后端后，后端立即产生一个 `Task_ID` 返回给前台，前台开始轮询（或通过 WebSocket 推送）。抓取回来的数据立刻写入 Redis 缓存并推送给前端展示，同时异步丢给后台 GPU 集群进行特征提取。

| 任务编号 | 任务描述 | 验收标准 |
|---------|---------|---------|
| 4.1.1 | 部署 Celery + Redis Broker | Worker 启动，可接收任务 |
| 4.1.2 | 封装 Milvus 检索异步任务 | 任务提交后可通过 Task_ID 查询进度 |
| 4.1.3 | 封装小红书接口调用异步任务 | 抓取结果写入 Redis 缓存 |
| 4.1.4 | 实现前端轮询/WebSocket 推送机制 | 前端 3 秒内收到首批结果 |

### 4.2 弹性降级算法（Constraint Relaxation Tree）

当初始的筛选条件请求返回空结果时，调度层必须执行确定性的降级算法。基于前端 LLM 赋予的 `elastic_weights`，构建一个最小优先队列（Min-Priority Queue）。每次降级，弹出权重最低（容忍度最高）的限制条件。对于数值型条件（如粉丝数），执行平滑扩展函数：`C_new = [C_min × 0.7, C_max × 1.3]`。降级过程的每一步都需要生成日志，推送给前端的 `<ElasticLoadingTerminal>` 组件实时展示。

| 任务编号 | 任务描述 | 验收标准 |
|---------|---------|---------|
| 4.2.1 | 实现最小优先队列构建逻辑 | 按 elastic_weights 正确排序 |
| 4.2.2 | 实现条件剥离与数值平滑扩展 | 降级后查询范围正确扩大 |
| 4.2.3 | 实现降级日志生成与推送 | 前端终端组件可实时渲染降级过程 |
| 4.2.4 | 集成到主检索流程 | 空结果时自动触发降级，最终返回结果 |

### 4.3 后端 API 层

基于 PRD 中定义的四个核心 API 契约，实现 RESTful 接口层。

| 任务编号 | 任务描述 | 对应 API |
|---------|---------|---------|
| 4.3.1 | 意图解析接口 | `POST /api/v1/intent/parse` |
| 4.3.2 | 弹性检索与反馈进化接口 | `POST /api/v1/match/retrieve` |
| 4.3.3 | 资产入库与打标接口 | `POST /api/v1/assets/commit` |
| 4.3.4 | 智能表头匹配与字典沉淀接口 | `POST /api/v1/export/confirm-mapping` |
| 4.3.5 | 达人资产库查询接口 | `GET /api/v1/library/list` |
| 4.3.6 | 履约历史查询接口 | `GET /api/v1/library/history` |

### Sprint 4 交付物

完成后应具备：通过 Postman 或 curl 调用所有 API，可完成"输入自然语言 → 返回排序达人列表 → 提交反馈 → 获取进化结果 → 确认入库"的完整后端闭环。

## Sprint 5：触点层 — 前端对接与 LLM 意图解析（预计 3 周）

本阶段将已完成的前端 UI（Σ.Match v1.0）与后端 API 进行对接，并集成 LLM 意图解析能力。

### 5.1 LLM 意图解析器集成

在后端实现 Few-shot CoT（少样本思维链）的 System Prompt，强制要求 LLM 输出强类型 JSON。输出结构需包含 `hard_filters`（转译为 Milvus 标量过滤条件）、`soft_vectors`（转化为 CLIP Text Embedding 的输入提示）、以及 `elastic_weights`（容忍度权重 1-5，指导降级策略）。前端不直接调用大模型，而是通过 `/api/v1/intent/parse` 接口间接调用。

| 任务编号 | 任务描述 | 验收标准 |
|---------|---------|---------|
| 5.1.1 | 编写 System Prompt 与 Few-shot 样本 | 10 条测试用例输出格式 100% 合规 |
| 5.1.2 | 实现 LLM 输出校验与容错 | JSON 解析失败时自动重试 |
| 5.1.3 | 将 soft_vectors 转化为 CLIP Embedding | 文本提示正确转为 768 维向量 |

### 5.2 前端 API 对接（替换 Mock 数据）

将 Σ.Match v1.0 中所有使用 Mock 数据的组件逐一替换为真实 API 调用。这是工作量最大但逻辑最清晰的部分，按页面维度拆分。

## Sprint 6：全链路联调、优化与上线（预计 2 周）

### 6.2 性能优化

| 任务编号 | 任务描述 | 目标指标 |
