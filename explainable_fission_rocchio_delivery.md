# 可解释 Fission 与精细化 Rocchio 交付说明

## 一、完成内容概述

本轮改造围绕两条主线继续推进。第一条是把前端 **意图确认面板** 与 **Fission 后的权重变化** 做得更可解释，让用户在点击“获取更多”后能直接看到系统到底把哪些标签升权、哪些标签降权，以及对应的权重变化幅度。第二条是把后端的 **Rocchio 初版** 升级为更精细的反馈进化策略，在下一批推荐中明确区分 **本轮反馈** 与 **历史反馈**，并支持结合 **操作角色** 做权重衰减与融合。

从当前落地结果看，前后端已经形成闭环：前端工作台能够请求真实的下一批推荐接口，后端能够返回更丰富的 `weight_changes` 与 `rocchio` 元数据，前端再把这些解释性信息回显为提示条、摘要卡片和弹窗中的最近一次调整说明。

## 二、本轮核心改动

| 模块 | 文件 | 本轮改动 |
|---|---|---|
| 前端页面 | `frontend/client/src/pages/Workspace.tsx` | 增加 Fission 权重变化提示条、Rocchio 反馈进化摘要卡、意图确认弹窗中的最近一次权重变化回显，并在下一批推荐请求中透传 `operator_role` |
| 前端 API 类型 | `frontend/client/src/lib/api.ts` | 为 `next-batch` 返回结果补充 `weight_changes`、`promoted/demoted`、`rocchio.breakdown`、`current_role`、`strategy` 等强类型结构 |
| 后端服务 | `backend/services/asset_service.py` | 已接入更细的 selected / rejected 反馈进化，并输出更可解释的权重升降信息与 Rocchio 分层元数据 |
| 后端契约 | `backend/api/schemas.py` | 扩展下一批推荐请求参数，支持 `operator_role`、本轮/历史反馈衰减参数与标签调整参数 |
| 后端测试 | `backend/tests/test_asset_services.py` | 新增并强化对权重变化解释、Rocchio 分层反馈、历史反馈介入与结果元数据结构的回归断言 |

## 三、前端可见效果

前端现在不再只是“调整了权重但用户不知道发生了什么”。本轮完成后，工作台新增了两层可解释反馈。

第一层是在 **Fission 之后的主工作台区域**，系统会显示一个显式提示区块。这里会告诉用户本轮有哪些标签被升权、有哪些标签被降权，并把每个标签的变化值直接展示成 `1.0 → 1.4` 这样的形式。与此同时，旁侧会展示 Rocchio 的反馈进化摘要，说明本轮正反馈、历史正反馈、本轮负反馈、历史负反馈各自参与了多少样本。

第二层是在 **意图确认弹窗** 中加入“最近一次 Fission 的权重变化”回显。如果用户在获取更多之后再次打开意图面板，就能直接看到刚刚系统调过哪些标签，而不是只看到一组已经变化过的滑块数值。

## 四、Rocchio 策略升级点

当前实现已经不再把所有反馈一股脑混在一起，而是开始按来源与角色拆分。整体上，后端会将反馈拆为 **本轮 selected / rejected** 与 **历史 selected / rejected** 两类，并为它们施加不同的基础衰减系数。与此同时，来自不同角色的历史任务也会在聚合时形成额外权重，使“客户、策划、采购”对推荐方向的影响可以在服务层继续精细化。

这意味着当前版本已经具备了向最终产品演进的关键骨架：不仅能做向量平移，还能把“谁在反馈、是当前轮还是历史轮、对哪些标签造成了影响”一起返回给前端解释层。

## 六、当前能力边界

本轮重点是把解释性和策略精细度往前推一大步，因此当前版本已经可以满足“用户能看见系统为什么改变权重”和“下一批推荐能区分本轮/历史反馈”这两个核心目标。

不过，当前前端回显仍然以 **标签级总结** 为主，还没有进一步细化到“某个标签的升权来自哪些具体 selected 达人、某个标签的降权来自哪些具体 rejected 达人”的证据卡片层。后端虽然已经提供了这类证据结构的基础，但前端暂时只消费了摘要部分。下一轮如果继续推进，可以把每个标签下方展开成“正反馈样本 / 负反馈样本”的证据列表，让解释性再提升一层。

## 七、建议的下一步

| 优先级 | 建议 | 目的 |
|---|---|---|
| P0 | 前端继续消费 `positive_examples` / `negative_examples` 证据字段 | 让用户不只看到升降权结果，还能看到触发这些变化的达人证据 |
| P0 | 在工作台中区分“本轮反馈驱动”和“历史反馈驱动”的标签变化来源 | 提升 Fission 解释的一致性，避免用户误以为所有变化都来自刚刚的选择 |
| P1 | 在角色权重上加入时间衰减 | 让较老的历史任务影响逐步减弱，更接近真实业务演化 |
| P1 | 将解释摘要沉淀到 `library/history` 视图中 | 让后续复盘能看到每一批次推荐是如何发生偏移的 |

## 九、结论

本轮改造已经把产品从“能做下一批推荐”推进到了“**能解释下一批推荐为什么这样变化**”。这对最终产品非常关键，因为用户从这一版开始，已经不仅仅是在使用一个黑盒推荐器，而是在和一个具备记忆、可解释、可逐轮校正的推荐工作台交互。
