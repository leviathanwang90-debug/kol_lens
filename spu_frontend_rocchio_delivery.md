# SPU 记忆前端接入、Fission 下一批推荐与 Rocchio 进化交付说明

## 一、 本轮目标与完成范围

本轮继续围绕你给出的四项优先级推进，重点不是再补一层原型说明，而是把 **SPU 推荐特征前端可见化**、**Fission 动作改接真实的 `match/next-batch`**、**后端增加 Rocchio 向量进化能力**、以及 **用户私有偏好记忆读取与参与推荐** 这四件事真正串起来。

当前已经落地的能力包含两个层面。第一层是后端：补齐了用户维度记忆接口，增强了下一批推荐编排，让它不再只是复用首轮请求，而是可以把 `selected / rejected` 反馈转成 Rocchio 风格的查询向量演化，再透传给检索服务。第二层是前端：工作台从原来的 Mock 流程改为真实请求链路，用户输入自然语言后，会先触发语义理解，再拉取 SPU 记忆和用户记忆，随后通过意图确认面板完成权重与指标调整，确认后才进入首轮真实检索；后续“获取更多”会直接走下一批推荐接口，而不是重复首轮查询。

## 三、 工作台前端的真实交互变化

当前 `/workspace` 已经不再是“伪流程演示”，而是能走通一条真实的产品主链路。

首先，用户输入自然语言后，前端会并行请求三类信息：意图解析结果、当前 SPU 的历史记忆、当前操作人的私有偏好记忆。随后弹出一层 **意图确认面板**，而不是直接开始搜索。这个面板已经包含你要求的几个关键结构：视觉/风格标签列表、每个标签独立的权重滑块、数据指标区间的可编辑输入项、SPU 推荐特征区块，以及用户私有标签区块。

其次，用户确认意图后，前端才会把“确认后的意图 + 标签权重 + 指标改写结果”发送给首轮检索接口。这样前端已经具备你要的“先格式化理解，再让用户复核和改写”的产品节奏。

再次，底部 Fission 动作现在已经改接真实的 `POST /api/v1/match/next-batch`。也就是说，当用户先在评审弹窗里做出 `selected / pending / rejected` 决策，再点击“获取更多”时，请求里会带上本轮反馈、SPU 记忆、用户记忆以及当前手动调整后的 tag 权重，由后端做真正的“下一批推荐”编排。

最后，“确认入库”动作也已经对接真实的 `POST /api/v1/assets/commit`。当前前端会把品牌、SPU、操作人、角色、确认后的意图、最终权重、数据需求、选中/淘汰/待定达人列表一起提交回后端。

## 四、 Rocchio 与用户私有偏好记忆是如何接入的

本轮后端不再把下一批推荐视为“首轮检索换个按钮名”，而是引入了更清晰的反馈进化路径。

具体来说，资产服务现在会先分别聚合 **SPU 维度历史任务** 与 **用户维度历史任务**。前者沉淀品牌/产品层面常见命中特征，后者沉淀当前操作人的筛选偏好。两者会分别产出推荐 tag 权重、选中/淘汰历史排行以及历史已看过 ID 集合。

在此基础上，下一批推荐流程会先构造基础查询向量，然后尝试从向量库中按达人 ID 读取本轮 `selected / rejected` 对应的真实向量。如果能够读到向量，就会按 Rocchio 方式对查询向量做正负反馈平移；如果当前会话反馈还不够，则可以回退使用历史反馈 ID 作为补充种子。最终，演化后的向量通过 `query_vector_override` 送入检索服务，形成真正意义上的“进化式 Fission”。

## 七、 建议你下一步继续做的两件事

第一件事，是继续把 **前端意图确认面板的权重变化回显** 做得更可解释。当前已经能调整权重，但还可以进一步在 Fission 返回后，用前端提示条明确告诉用户：“系统把哪些 tag 从 1.0 调到 1.4、哪些标签被降权”，这样更贴近你最早设想的产品反馈感。

第二件事，是继续把 **Rocchio 初版升级为更精细的 selected / rejected 向量进化策略**。当前已经有后端能力基础，下一步就可以开始区分“本轮反馈”和“历史反馈”的权重，甚至引入不同角色（客户、策划、采购）的反馈权重衰减规则，让推荐质量继续向最终产品设计靠近。
