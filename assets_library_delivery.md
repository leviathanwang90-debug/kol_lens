# kol_lens 资产提交与资产库接口交付说明

## 本轮完成内容

本轮按照你重新设定的优先级，先完成了三组产品级接口的后端落地：`assets/commit`、`library/list`、`library/history`。这三组接口的目标，是把当前已经能完成的“找人”主链路，继续往“评审后确认入库、进入资产库、查询历史沉淀”的正式产品闭环推进。

| 文件 | 作用 |
|---|---|
| `backend/services/asset_service.py` | 新增资产服务层，封装资产提交、资产库列表查询、资产历史查询 |

## 新增接口说明

### 1. `POST /api/v1/assets/commit`

这个接口用于承接前端评审后的“确认入库”动作。它支持传入 `brand_name`、`spu_name`、`selected_ids`、`rejected_ids`、`pending_ids`、`tag_weights`、`data_requirements`、`query_vector` 等字段。

当前行为是：如果前端还没有已有 `campaign_id`，接口会先创建一条寻星任务；然后把本轮选中、淘汰、待定结果写回 `campaign_history`；再往 `fulfillment_records` 写入一条履约/提交记录。这样后面无论做 SPU 偏好沉淀、换一批进化，还是资产库历史回看，都已经有结构化数据基础。

它返回的结果里，额外带了一个 `next_batch_strategy` 字段，明确标识当前 SPU 的反馈记忆已经沉淀，可作为后续“下一批推荐”的基础。

### 2. `GET /api/v1/library/list`

这个接口用于资产库列表页。当前支持按 `region`、`followers_min`、`followers_max`、`gender`、`tags`、`sort_by`、`sort_order`、分页参数来筛选和排序达人。

返回结构里包含 `items` 和 `pagination`，并在每个达人上补了一个 `history_hint.detail_path`，方便前端直接跳到该达人的历史接口。

### 3. `GET /api/v1/library/history`

这个接口当前支持三种查询模式：

| 模式 | 入参 | 返回内容 |
|---|---|---|
| 达人历史 | `influencer_id` | 返回该达人跨任务的履约历史 |
| 任务时间轴 | `campaign_id` | 返回该任务的履约/提交时间轴 |
| 品牌/SPU 历史 | `brand_name` + 可选 `spu_name` | 返回该品牌或 SPU 的历史 campaign 列表 |

这样做的好处是前端的资产库页、达人详情抽屉、品牌/SPU 历史面板都可以先复用同一个历史接口，再决定 UI 层如何拆分展示。

## 关于你提出的前端交互设计

你描述的这套前端交互，我认为是合理的，而且**应该作为最终产品的标准交互层**来设计，不建议只把自然语言解析结果简单展示成一段 JSON。

### 我建议的前端确认弹窗结构

在用户输入自然语言、完成第一次语义理解后，前端确实应该弹出一个“意图确认弹窗”。这个弹窗的核心目标，不是给用户看模型过程，而是把模型输出转成**可编辑、可控、可继续优化的检索参数面板**。

建议结构如下。

| 区域 | 展示内容 | 交互方式 |
|---|---|---|
| 风格化标签区 | 画面气质、人设感觉、场景类型、文案感、商业感等 tag | 每个 tag 下方滑块，范围 `0~2`，默认 `1` |
| 数据指标区 | 粉丝范围、图文报价、视频报价、CPM、目标人数等 | 直接编辑区间或数值 |
| 检索策略区 | 是否启用扩库、是否启用降级、目标返回数 | 开关或数字输入 |
| 摘要区 | 长句 query、格式化标签摘要 | 只读，用于帮助用户理解本轮检索语义 |

这套结构和当前后端也已经基本契合。因为现在 `intent/parse` 会返回 `data_requirements` 和 `query_plan.formatted_tags`，`match/retrieve` 又已经支持 `tag_weights` 与改写后的数据指标继续传入。因此后端层面已经能承接这套交互，不需要推翻重做。

## 关于“用户选中/未选中后，是否应该有一个针对当前 SPU 的库，用于推荐下一批博主”

我的判断是：**应该有，但它不应该被设计成一套新的独立达人库，而应该被设计成“SPU 级偏好记忆层”**。

也就是说，下一批推荐不应简单理解为“再建一个 SPU 的达人表”，而应该理解为：

> 针对当前 `brand + spu`，沉淀该 SPU 在历次任务中的意图快照、查询向量、选中集合、淘汰集合、待定集合，以及必要时的角色权重，然后基于这些记忆去生成下一轮推荐向量和过滤偏好。

这层东西，本质上更像是**SPU 级推荐记忆库**，而不是资产库的副本。它至少应该沉淀以下内容：

| 沉淀对象 | 用途 |
|---|---|
| 原始意图与改写后意图 | 还原当时为什么这么找 |
| 调整后的 tag 权重 | 记录该 SPU 的风格偏好 |
| 数据指标改写结果 | 记录业务真实可接受范围 |
| selected / rejected / pending | 为下一批进化提供监督信号 |
| query_vector_snapshot | 为 Rocchio 或其他进化算法提供起点 |
| operator_role | 区分客户、策划、采购的反馈权重 |

从这个角度看，本轮新增的 `assets/commit` 已经开始为这件事打地基：它把 `campaign_history + fulfillment_records` 接通了，并且保留了 `query_vector`、反馈列表和意图快照。后面你要做“下一批推荐”，最合适的做法就是在这套 SPU 记忆层之上实现进化检索，而不是再建一套独立物理库。

## 我建议你后续的实现顺序

如果沿着你现在的产品方向继续推进，我建议后续顺序如下。

| 优先级 | 建议 | 原因 |
|---|---|---|
| P0 | 让前端接入意图确认弹窗，支持 tag 权重滑块和数据指标改写 | 这是用户真正可控的第一层体验 |
| P0 | 在 `assets/commit` 基础上补“SPU 级偏好记忆”读取接口 | 为下一批推荐做准备 |
| P1 | 在 `match/retrieve` 中接入基于 SPU 历史反馈的进化推荐 | 完成“换一批”核心价值 |
| P1 | 增加角色权重沉淀策略 | 让客户/策划/采购反馈产生不同影响 |
| P2 | 再接入 Rocchio 或更完整的反馈向量平移 | 把推荐从规则增强升级为真正的进化引擎 |
